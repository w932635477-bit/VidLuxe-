## 2. Autonomous TDD 深度分析

### 优势

| 优势 | 说明 | 实际效果 |
|------|------|----------|
| **完全自动化** | 写完测试后无需人工干预 | 节省 60-80% 开发时间 |
| **测试先行** | 保证代码质量 | 减少 70% 回归 Bug |
| **自我修正** | 失败后自动换方案 | 提高成功率 |
| **可追溯** | 记录所有尝试过程 | 便于事后分析 |

### 劣势

| 劣势 | 原因 | 缓解策略 |
|------|------|----------|
| **可能走偏** | 理解需求偏差 | 提供详细需求描述 |
| **迭代耗时** | 多次尝试修复 | 设置合理超时 |
| **测试质量依赖** | 测试写不好=实现不好 | 覆盖全面场景 |
| **复杂逻辑受限** | 难以处理复杂业务 | 拆分为小功能 |

### 专业使用技巧

#### ✅ 适合使用的场景

```
1. 功能边界明确的新功能
   例如：实现用户登录、文件上传、数据导出

2. 有明确输入输出的 API
   例如：/api/payment/create - 输入：订单信息，输出：支付链接

3. 可复现的 Bug
   例如：视频上传后音频丢失 - 写测试复现，自动修复

4. 算法实现
   例如：图片压缩、帧提取、数据转换
```

#### ❌ 不适合使用的场景

```
1. 需要创意设计的功能
   例如：UI 布局、动画效果、用户交互

2. 高度依赖外部服务的功能
   例如：第三方支付回调、微信授权

3. 需要人工判断的功能
   例如：内容审核、图片质量评分

4. 跨多个系统的复杂集成
   例如：整个支付流程（前端+后端+第三方）
```

### 实际案例对比

#### ❌ 差的需求描述

```
/autonomous-tdd "实现一个支付功能"
```

**问题**：太模糊，Agent 无法知道：
- 支持什么支付方式？
- 需要什么参数？
- 返回什么结果？
- 错误如何处理？

#### ✅ 好的需求描述

```
/autonomous-tdd "

功能：创建微信支付订单 API

API 路径：POST /api/payment/create

输入：
- packageId: string - 积分包ID（'small' | 'medium' | 'large'）
- userId: string - 用户ID
- payType: 'native' | 'jsapi' - 支付类型
- openid?: string - JSAPI 支付需要的用户 openid

输出：
- success: boolean
- codeUrl?: string - Native 支付的二维码链接
- miniProgramPayParams?: object - 小程序支付参数
- error?: string - 错误信息

业务规则：
1. packageId 必须是有效的积分包
2. Native 支付返回 codeUrl
3. JSAPI 支付需要 openid，返回 miniProgramPayParams

错误处理：
- 无效 packageId：'套餐不存在'
- JSAPI 缺少 openid：'JSAPI 支付需要用户 openid'
- 微信支付未配置：'微信支付未配置'

技术约束：
- 使用 /lib/wechat-pay/client.ts 中的 createNativeOrder 和 createJSAPIOrder
"
```

## 3. Parallel Refactor 深度分析

### 优势

| 优势 | 说明 | 实际效果 |
|------|------|----------|
| **并行执行** | 多个 Agent 同时工作 | 4倍速度提升 |
| **架构分层** | 按层次分配任务 | 减少冲突 |
| **依赖协调** | 自动处理跨层依赖 | 避免破坏性修改 |
| **进度可视** | 实时追踪进度 | 清晰的完成状态 |

### 劣势

| 劣势 | 原因 | 缓解策略 |
|------|------|----------|
| **规划复杂** | 需要提前分析依赖 | 详细描述迁移范围 |
| **协调成本** | Agent 间可能冲突 | 使用文件声明机制 |
| **适合大型任务** | 小任务反而慢 | 仅用于 >10 文件的变更 |
| **需要回滚准备** | 大规模修改风险高 | 保留备份分支 |

### 专业使用技巧

#### ✅ 适合使用的场景

```
1. 模式迁移（10+ 文件）
   例如：Redux → Zustand，涉及 5 个 store 和 20 个组件

2. API 契约变更
   例如：REST API 响应格式从 v1 升级到 v2

3. 命名/结构重构
   例如：重命名所有以 'style' 开头的变量为 'effect'

4. 框架升级适配
   例如：React 17 → React 18，修改所有组件
```

#### ❌ 不适合使用的场景

```
1. 单文件或少量文件修改
   直接用 Edit 工具更快

2. 逻辑复杂的重构
   需要深入理解业务，Agent 可能改错

3. 没有测试覆盖的代码
   无法验证修改是否正确

4. 紧急修复
   规划和协调需要时间
```

#### 🎯 提高成功率的技巧

```markdown
## 迁移计划模板

/parallel-refactor "

## 迁移目标
[旧模式] → [新模式]

## 受影响范围

### Store 层（Agent A）
- store/user.ts
- store/video.ts
- store/image.ts

### Component 层（Agent B）
- components/UserList.tsx（依赖 store/user.ts）
- components/VideoPlayer.tsx（依赖 store/video.ts）

### API 层（Agent C）
- api/user/route.ts
- api/video/route.ts

### Test 层（Agent D）
- __tests__/user.test.ts
- __tests__/video.test.ts

## 依赖关系
1. Component 层必须等待 Store 层完成
2. Test 层最后执行
3. API 层可以与 Store 层并行

## 迁移规则
- [旧代码模式] → [新代码模式]
- 例如：useSelector(state => state.user) → useUserStore()

## 验证方式
- 每个 Agent 完成后运行相关测试
- 全部完成后运行 pnpm test
"
```

## 4. Deploy Checklist 深度分析

### 优势

| 优势 | 说明 | 实际效果 |
|------|------|----------|
| **标准化流程** | 每次部署一致 | 减少人为失误 |
| **问题早发现** | 部署前检查 | 避免上线后故障 |
| **可回滚** | 保留备份 | 快速恢复 |
| **文档化** | 检查记录可追溯 | 便于排查问题 |

### 劣势

| 劣势 | 原因 | 缓解策略 |
|------|------|----------|
| **步骤固定** | 不够灵活 | 根据项目调整 |
| **手动执行** | 部分步骤需人工 | 结合 CI/CD 自动化 |
| **时间成本** | 检查需要时间 | 关键步骤必检，其他可选 |

### 专业使用技巧

#### ✅ 必须检查的项目

```bash
# 1. 类型检查（必须）
pnpm typecheck

# 2. 构建（必须）
pnpm build

# 3. 服务器磁盘空间（必须）
ssh root@server "df -h"
# 使用率 > 80% 必须清理

# 4. 部署后健康检查（必须）
curl https://your-domain/api/health
```

#### ⚠️ 可选检查的项目

```bash
# 1. 单元测试（推荐）
pnpm test

# 2. E2E 测试（重要变更时）
pnpm test:e2e

# 3. 服务器内存（高负载时）
ssh root@server "free -h"

# 4. 日志检查（有报错时）
ssh root@server "pm2 logs --lines 50"
```

### 常见问题快速处理

| 问题 | 检测命令 | 解决方案 |
|------|----------|----------|
| 磁盘满 | `df -h` | `docker system prune -af` |
| 构建失败 | `pnpm build` | 检查错误信息，修复后重试 |
| 502 错误 | `pm2 status` | `pm2 restart vidluxe` |
| 内存不足 | `free -h` | 重启服务或扩容 |
| DNS 未更新 | `dig @8.8.8.8` | 等待传播或清除缓存 |

## 5. 选择决策树

### 快速选择指南

```
开始
  │
  ├─ 是否涉及部署？
  │   ├─ 是 ──▶ /deploy-checklist
  │   └─ 否 ──▶ 继续
  │
  ├─ 是否修改 >10 个文件？
  │   ├─ 是 ──▶ /parallel-refactor
  │   └─ 否 ──▶ 继续
  │
  ├─ 是否是新功能或可复现的 Bug？
  │   ├─ 是 ──▶ /autonomous-tdd
  │   └─ 否 ──▶ 手动处理
  │
  └─ 是否需要创意或人工判断？
      ├─ 是 ──▶ 手动处理
      └─ 否 ──▶ /autonomous-tdd
```

## 6. 最佳实践组合

### 组合使用策略

#### 场景 1：新功能开发上线

```
1. /autonomous-tdd    ← 实现功能（自动迭代）
2. /deploy-checklist  ← 部署前检查（确保安全）
3. 部署到生产环境
```

#### 场景 2：大规模重构

```
1. /parallel-refactor ← 并行重构（多 Agent 协作）
2. /deploy-checklist  ← 部署前检查
3. 部署到生产环境
```

#### 场景 3：紧急 Bug 修复

```
1. /autonomous-tdd    ← 写测试复现 Bug，自动修复
2. 快速部署（跳过部分检查）
3. 事后补充测试
```

#### 场景 4：日常维护

```
1. /deploy-checklist  ← 常规部署检查
2. 小改动手动处理
3. 大改动用 TDD
```

## 总结

### 核心原则

1. **选对工具** - 根据任务特点选择合适的 Skill
2. **详细描述** - 给 Agent 提供充分的信息
3. **验证结果** - 不要盲目信任，检查输出
4. **保留回滚** - 大改动前做好备份

### 效率提升建议

| Skill | 效率提升点 | 关键成功因素 |
|-------|-----------|--------------|
| TDD | 需求描述详细度 | 测试覆盖全面 |
| Parallel | 依赖分析准确 | 分层合理 |
| Checklist | 标准化流程 | 每步确认 |
